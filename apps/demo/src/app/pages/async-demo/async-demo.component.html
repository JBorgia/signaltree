Async Markers

The SignalTree-native way to model async state. asyncSource and asyncQuery attach at any tree path and auto-expose data, loading, error, and lifecycle methods — same family as entityMap, status, stored, and form. 

import { asyncSource, asyncQuery } from '@signaltree/core';

asyncSource — load-and-expose

Define a loader once. The marker materializes into a fully-functional async accessor: data, loading, error, refresh — all derived automatically. No manual .setLoading() / .setLoaded() wiring, no tap() ceremony in a pipeline.

const store = signalTree({
  users: asyncSource<User[]>({
    initial: [],
    load: () => this.api.list$(),  // returns Observable<User[]>
  }),
});

// Read:
store.$.users();           // User[] | undefined
store.$.users.loading();   // boolean
store.$.users.error();     // unknown | null

// Drive lifecycle:
store.$.users.refresh();   // reload (cancels in-flight)
store.$.users.set([...]);  // manual override
store.$.users.reset();     // clear data/error/loading

asyncQuery — input-driven debounced query

Bind an input signal; every change pipes through the debounce → filter → distinctUntilChanged → switchMap(query) pipeline automatically. Race conditions, cancellation, deduplication — all handled.

@if (store.$.search.error()) { ⚠️ Error } @else if (store.$.search.loading()) { ⏳ searching… } @else if ((store.$.search()?.length ?? 0) > 0) { ✅ {{ store.$.search()!.length }} matches } @else if (store.$.search.input()) { ⚪ no matches } @else { ⚪ idle }

const store = signalTree({
  search: asyncQuery<string, User[]>({
    initialResult: [],
    debounce: 300,
    filter: (q) => q.length > 0,
    query: (q) => this.api.search$(q),
  }),
});

// Push input — pipeline fires after debounce + dedup:
store.$.search.input.set('alice');

// Or two-way bind in the template:
<input [(ngModel)]="store.$.search.input">

// Read results:
store.$.search();          // results
store.$.search.loading();  // in-flight
store.$.search.error();

What you're seeing

  • Both markers attach at any depth. Here they live at depth 1 ($.users, $.search), but you can drop them anywhere in your state literal — same as entityMap, status, stored, and form.
  • No manual status wiring. The marker materializer creates loading/error signals and toggles them as the load runs.
  • Auto-cleanup. Subscriptions tear down on the surrounding DestroyRef (component / service teardown).
  • Observable OR Promise. Both load() and query() accept either return type — no firstValueFrom ceremony.

Coming from NgRx?

rxMethod remains available at @signaltree/core/rxjs-interop as a migration-friendly alias. New SignalTree code should prefer the asyncSource / asyncQuery markers — they compose with the rest of the marker family and eliminate the manual tap()/setLoading() wiring rxMethod requires.
