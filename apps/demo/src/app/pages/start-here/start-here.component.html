Step 1 · ~30 seconds

State is data. Reads call. Writes call.

A tree is initialized with a plain JSON object. Every leaf is a typed Angular signal — you read it by calling and write with .set or .update. Every branch (any non-leaf node) is callable too: call it to read the subtree, or call it with a partial to deep-merge a write — no transform required. There are no actions, reducers, selectors, or registration steps. Path-based access stays type-safe at any depth. 

{{ mentalModelCode }}

That's the entire mental model. Everything else — entity collections, DevTools, persistence, time travel, forms, realtime — composes on top with .with(...) enhancers.

Step 3 · ~1 minute

One runtime tree, typed slices, root-level enhancers.

The recommended pattern: a single tree per app, sliced by feature. DevTools, persistence, and time travel attach at the root, not per feature. Components consume typed slices directly. 

{{ architectureCode }}

Why one tree? A single source of truth: one DevTools panel, one time-travel history, one persistence layer. As the app grows, the pattern that keeps it consistent is READ through $, WRITE through ops, REACT through effect() — see recommended architecture for the full pattern, live.
