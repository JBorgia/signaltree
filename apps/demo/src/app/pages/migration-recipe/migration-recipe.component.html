Migrating to SignalTree

SignalTree is designed to be a drop-in replacement for the most common Angular state-management patterns. Pick your starting point below — each section maps the source library's concepts onto SignalTree, shows side-by-side code, and links to the deep guide your AI assistant can follow end-to-end.

@ngrx/signals (signalStore)

The closest relative — both libraries give you typed signals over a state object. SignalTree replaces the withX composition model with a single root tree and plain Ops services.

Concept map

@ngrx/signalsSignalTree
signalStore({ providedIn: 'root' }, ...)Tree slice on AppTree + sibling Ops service
withState<T>(initial)Initial state passed to signalTree(initial)
withComputed(store => ({...}))computed() on the tree, or a Computed service
withMethods(store => ({...}))Methods on an Ops service that update the tree
withHooks({ onInit })Factory body or Ops constructor
rxMethod(source$ => ...)Plain method, optionally returning Observable<void>
withEntities<Entity>()entityMap<Entity>() marker
patchState(store, partial)tree.$.field.set(value) (leaf); tree.$.obj.update(s => ({...s, ...patch})) (object slice)
getState(store)tree.$.slice() — no-arg call returns current value
withReduxDevtools()devTools() enhancer

Side-by-side

Before — @ngrx/signals

export const TicketStore = signalStore(
  { providedIn: 'root' },
  withState<TicketState>({ tickets: [], filter: '' }),
  withComputed((s) => ({
    visible: computed(() =>
      s.tickets().filter(t => t.title.includes(s.filter()))
    )
  })),
  withMethods((s, api = inject(TicketApi)) => ({
    load: rxMethod<void>(pipe(
      switchMap(() => api.list()),
      tap(tickets => patchState(s, { tickets }))
    ))
  }))
);

After — SignalTree

// app.tree.ts — single root tree
export const APP_TREE = new InjectionToken('AppTree', {
  factory: () => signalTree({
    tickets: entityMap<Ticket>(),
    filter: '',
  })
});

// ticket.ops.ts — plain service
@Injectable({ providedIn: 'root' })
export class TicketOps {
  private tree = inject(APP_TREE);
  private api = inject(TicketApi);

  visible = computed(() =>
    this.tree.$.tickets.all().filter(t =>
      t.title.includes(this.tree.$.filter())
    )
  );

  async load() {
    const tickets = await firstValueFrom(this.api.list());
    this.tree.$.tickets.setAll(tickets);
  }
}

Why two files? For a single simple store, SignalTree is slightly more lines. The split pays off at scale: the tree stays one file regardless of how many domains you add, and each Ops service is a plain @Injectable with no state-management abstractions — easier to test in isolation, easier to tree-shake, no custom withX helpers to maintain. At 5+ stores, NgRx SignalStore grows 5+ store files each with custom features; SignalTree adds 5 Ops files and zero new concepts.

Full playbook: the deep migration guide with three strategies (big-bang, incremental, hybrid), a Phase 0 recipe, sequencing rules, and a verifier script lives at docs/skills/using-signaltree/reference/migration-from-ngrx-signals.md. It also ships inside @signaltree/core as an Agent Skill — see AI-assisted migration.

Classic NgRx (@ngrx/store + @ngrx/effects)

The actions / reducers / selectors / effects model collapses into a single tree plus async methods on an Ops service. Most "type the action, dispatch the action, switch on the action" boilerplate simply disappears.

NgRxSignalTree
createAction + props<T>()Method parameters on an Ops service
createReducer + on(...)Direct tree updates inside the method
createSelector / createFeatureSelectorcomputed() over tree.$.slice
createEffect + switchMap / concatMapasync method on Ops, or rxMethod-style helper
EntityAdapter / createEntityAdapterentityMap<T>() marker
Store.dispatch(action)ops.method(args)
store.select(selector)tree.$.path or a computed()
StoreDevtoolsModuledevTools() enhancer

Before — NgRx

// actions
export const loadUsers = createAction('[Users] Load');
export const loadUsersSuccess = createAction(
  '[Users] Load Success', props<{ users: User[] }>()
);

// reducer
export const usersReducer = createReducer(initialState,
  on(loadUsersSuccess, (s, { users }) => ({ ...s, users }))
);

// effect
loadUsers$ = createEffect(() => this.actions$.pipe(
  ofType(loadUsers),
  switchMap(() => this.api.list().pipe(
    map(users => loadUsersSuccess({ users }))
  ))
));

// component
users$ = this.store.select(selectUsers);
this.store.dispatch(loadUsers());

After — SignalTree

// users.ops.ts
@Injectable({ providedIn: 'root' })
export class UsersOps {
  private tree = inject(APP_TREE);
  private api = inject(UserApi);

  async load() {
    this.tree.$.users.loading.set(true);
    try {
      const users = await firstValueFrom(this.api.list());
      this.tree.$.users.entities.setAll(users);
    } finally {
      this.tree.$.users.loading.set(false);
    }
  }
}

// component
users = this.tree.$.users.entities.all;
ops = inject(UsersOps);
// ops.load()

What this comparison shows: the NgRx side displays the complete boilerplate required across typically three or more files — actions, reducer, effect, and component call-site. The SignalTree side consolidates all of that into a single Ops method. The APP_TREE token and tree definition (shared across all features) are not shown; see the architecture guide for setup.

Strategy: migrate one feature module at a time. Build the AppTree + a feature Ops service alongside the existing reducer, switch the components, then delete the actions / reducer / effects / selectors files in the same PR.

@ngrx/component-store

Component Store is closest in shape — local state, updaters, effects. Map each ComponentStore to an Ops service bound to a tree slice (or to a small per-component signalTree() if it really is component-scoped).

@ngrx/component-storeSignalTree
extends ComponentStore<T>Tree slice + Ops service
this.updater((s, v) => ...)Method that calls tree.$.x.update(...)
this.select(...)computed() over the tree
this.effect<T>(source$ => ...)async method, or RxJS pipeline returning Observable<void>
this.patchState(partial)tree.$.field.set(value) per leaf, or tree.$.obj.update(s => ({...s, ...patch}))

NGXS

NGXS state classes with @Action handlers and @Selector map directly onto an Ops service with plain methods plus computed() selectors.

NGXSSignalTree
@State<T>({ name, defaults })Tree slice on AppTree
@Selector()computed() on the Ops service
@Action(MyAction)Method on the Ops service
ctx.patchState(partial)tree.$.field.set(value) per leaf, or tree.$.obj.update(s => ({...s, ...patch}))
ctx.dispatch(new Other())Direct call to another Ops method
store.select(MyState.x)tree.$.x signal
NgxsReduxDevtoolsPluginModuledevTools() enhancer

Note: the action-class registry that NGXS uses for cross-state dispatch becomes a normal Angular dependency in SignalTree — OpsA just inject()s OpsB and calls its method.

Elf / Akita

Elf's repository pattern (and its predecessor Akita's EntityStore / QueryEntity) collapse into entityMap() markers on the tree plus computed selectors.

store.update(deleteEntities(id))tree.$.entities.removeOne(id)

@rx-angular/state

RxState exposes get, set, select, and connect over a flat object. The tree replaces the object; computed() replaces select; and connect's "merge an Observable into state" pattern becomes an explicit subscribe in the Ops constructor (or toSignal()).

@rx-angular/stateSignalTree
state.set(partial)tree.$.field.set(value) per leaf, or tree.$.obj.update(s => ({...s, ...patch}))
state.get('key')tree.$.key()
state.select('key')tree.$.key signal (or toObservable(...))
state.connect('key', source$)source$.subscribe(v => tree.$.key.set(v)) in Ops constructor

Plain BehaviorSubject services

The classic private _x$ = new BehaviorSubject<T>(initial) pattern becomes a tree slice plus a thin Ops service. You keep the mental model — services own the state — and gain typed paths, computed(), DevTools, and time travel for free.

Before

@Injectable({ providedIn: 'root' })
export class CartService {
  private _items$ = new BehaviorSubject<Item[]>([]);
  items$ = this._items$.asObservable();
  count$ = this.items$.pipe(map(xs => xs.length));

  add(item: Item) {
    this._items$.next([...this._items$.value, item]);
  }
}

After

@Injectable({ providedIn: 'root' })
export class CartOps {
  private tree = inject(APP_TREE);

  items = this.tree.$.cart.items;            // signal<Item[]>
  count = computed(() => this.items().length);

  add(item: Item) {
    this.tree.$.cart.items.update(xs => [...xs, item]);
  }
}

Plain Angular signals + service

If you already have signal() + computed() inside an @Injectable, you mostly just relocate the state into the shared tree. The wins are: one place to inspect everything in DevTools, time travel, persistence, and a single well-typed graph instead of N independent service-owned signals.

When you might not migrate: truly local UI state (collapsed/expanded, hover, transient form state) can stay as plain signal() in the component. The tree is for application state.

AI-assisted migration

SignalTree ships a vendor-neutral Agent Skill that lets Cursor, Claude Code, or any SKILL.md-aware AI assistant run these migrations end-to-end. After installing @signaltree/core the skill lives at node_modules/@signaltree/core/skills/using-signaltree/.

Install the skill once per repo

# Cursor
cp -r node_modules/@signaltree/core/skills/using-signaltree .cursor/skills/

# Claude Code
cp -r node_modules/@signaltree/core/skills/using-signaltree .claude/skills/

What the skill gives the agent

  • Mechanical concept maps for every source library listed above (the deepest coverage is currently @ngrx/signals; PRs welcome for the others).
  • Three migration strategies with explicit decision criteria — big-bang (one PR), incremental per-domain (one PR per store, with a Phase 0 foundation PR and a sequencing rule), and a hybrid legacy-facade for regulated rollback.
  • A drop-in verifier script (scripts/verify-signaltree-migration.sh in the SignalTree repo) that runs build + test + lint and asserts the source library is gone from application code and package.json. Package-manager and layout agnostic.
  • Anti-pattern catalog (root-injected Ops with eager side-effects, multiple competing trees, brown-field APP_TREE placement) so the agent doesn't repeat known traps.

Point your AI assistant at the skill, give it a target store to migrate, and it will follow the playbook and run the verifier itself. See Using SignalTree with AI Agents for harness-specific setup.
