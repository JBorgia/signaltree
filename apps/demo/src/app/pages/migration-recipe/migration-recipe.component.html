Migrating to SignalTree

SignalTree is designed to be a drop-in replacement for the most common Angular state-management patterns. This guide is structured feature-first: each scenario below shows the SAME requirement implemented in SignalTree, NgRx SignalStore, Classic NgRx, and a plain BehaviorSubject service, anchored against an explicit capability checklist. If a checkbox would be empty on either side, it isn't a fair comparison.

Why feature-first? Library-anchored before/after comparisons tempt the author to strip the "before" to its minimum and polish the "after" to its end state. Anchoring on a feature with a checklist prevents that bias — every implementation must satisfy every checkbox or it isn't a fair comparison.

Feature scenarios (full implementations):
  • Feature 1: Load + filter + loading state
  • Feature 2: Optimistic update with rollback
  • Feature 3: Reactive side effect on state change
  • Feature 4: Async load with error state + retry
Library concept maps (quick reference):
  • @ngrx/signals (signalStore)
  • Classic NgRx (@ngrx/store + effects)
  • @ngrx/component-store
  • NGXS
  • Elf / Akita
  • @rx-angular/state
  • Plain BehaviorSubject services
  • Plain Angular signals + service
  • AI-assisted migration

Feature 1: Load tickets with filter + loading state

The same feature, implemented four ways. Each implementation must satisfy every checkbox below — that's how the comparison stays honest.

Capability checklist — every implementation must:
  • Hold filter (string) and loading (boolean) state
  • Hold tickets as a normalized entity collection (id-keyed lookup, not just a flat array)
  • Expose a visible derived value (tickets filtered by title contains filter)
  • Provide async load() that toggles loading around the API call
  • Recover loading state on error (loading goes back to false on failure too)
  • Singleton, registered at root DI scope
  • Show every file the feature requires — no hidden selectors, no hidden registrations, no hidden types

1.1 SignalTree

Files: app.tree.ts (state + derived), app.config.ts (provider), ticket.ops.ts (writes), component. 

// app.tree.ts — state + all derived reads
export const APP_TREE = new InjectionToken<ReturnType<typeof createAppTree>>('APP_TREE');

export function createAppTree() {
  return signalTree({
    tickets: entityMap<Ticket>(),
    filter: '',
    loading: false,
  }).derived(($) => ({
    tickets: {
      visible: computed(() =>
        $.tickets.all().filter((t) => t.title.includes($.filter()))
      ),
    },
  }));
}

// app.config.ts — root provider registration
export const appConfig: ApplicationConfig = {
  providers: [{ provide: APP_TREE, useFactory: createAppTree }],
};

// ticket.ops.ts — async writes only
@Injectable({ providedIn: 'root' })
export class TicketOps {
  private tree = inject(APP_TREE);
  private api = inject(TicketApi);

  async load() {
    this.tree.$.loading.set(true);
    try {
      const tickets = await firstValueFrom(this.api.list());
      this.tree.$.tickets.setAll(tickets);
    } finally {
      this.tree.$.loading.set(false);
    }
  }

  setFilter(filter: string) {
    this.tree.$.filter.set(filter);
  }
}

// tickets-list.component.ts
@Component({ selector: 'app-tickets', template: '...' })
export class TicketsListComponent {
  private tree = inject(APP_TREE);
  ops = inject(TicketOps);

  visible = this.tree.$.tickets.visible;   // Signal<Ticket[]>
  loading = this.tree.$.loading;           // Signal<boolean>
  filter = this.tree.$.filter;             // Signal<string>

  ngOnInit() { this.ops.load(); }
}

1.2 NgRx SignalStore (@ngrx/signals)

Files: tickets.store.ts, component. providedIn: 'root' auto-registers — no separate provider. 

// tickets.store.ts — state + reads + writes in one factory
export const TicketStore = signalStore(
  { providedIn: 'root' },
  withState({ filter: '', loading: false }),
  withEntities<Ticket>(),
  withComputed((store) => ({
    visible: computed(() =>
      store.entities().filter((t) => t.title.includes(store.filter()))
    ),
  })),
  withMethods((store, api = inject(TicketApi)) => ({
    load: rxMethod<void>(
      pipe(
        tap(() => patchState(store, { loading: true })),
        switchMap(() =>
          api.list().pipe(
            tap((tickets) =>
              patchState(store, setAllEntities(tickets), { loading: false })
            ),
            catchError(() => {
              patchState(store, { loading: false });
              return EMPTY;
            })
          )
        )
      )
    ),
    setFilter(filter: string) {
      patchState(store, { filter });
    },
  }))
);

// tickets-list.component.ts
@Component({ selector: 'app-tickets', template: '...' })
export class TicketsListComponent {
  store = inject(TicketStore);

  visible = this.store.visible;     // Signal<Ticket[]>
  loading = this.store.loading;     // Signal<boolean>
  filter = this.store.filter;       // Signal<string>

1.4 Plain BehaviorSubject service

Files: ticket.service.ts, component. Note: RxJS doesn't ship entity normalization — you write the Map<string, Ticket> yourself to match what the other libraries provide. 

// ticket.service.ts
@Injectable({ providedIn: 'root' })
export class TicketService {
  // Manual normalization: ids list + by-id map
  private _byId = new Map<string, Ticket>();
  private _ids$ = new BehaviorSubject<string[]>([]);
  private _filter$ = new BehaviorSubject<string>('');
  private _loading$ = new BehaviorSubject<boolean>(false);

  filter$ = this._filter$.asObservable();
  loading$ = this._loading$.asObservable();

  // Derived: full list rebuilt from ids + map
  all$ = this._ids$.pipe(
    map((ids) => ids.map((id) => this._byId.get(id)!).filter(Boolean))
  );

  visible$ = combineLatest([this.all$, this.filter$]).pipe(
    map(([tickets, filter]) =>
      tickets.filter((t) => t.title.includes(filter))
    )
  );

  constructor(private api: TicketApi) {}

  byId(id: string): Ticket | undefined {
    return this._byId.get(id);
  }

  setFilter(filter: string) {
    this._filter$.next(filter);
  }

  async load() {
    this._loading$.next(true);
    try {
      const tickets = await firstValueFrom(this.api.list());
      this._byId.clear();
      tickets.forEach((t) => this._byId.set(t.id, t));
      this._ids$.next(tickets.map((t) => t.id));
    } finally {
      this._loading$.next(false);
    }
  }
}

// tickets-list.component.ts
@Component({ selector: 'app-tickets', template: '...' })
export class TicketsListComponent {
  service = inject(TicketService);

  visible = toSignal(this.service.visible$, { initialValue: [] });
  loading = toSignal(this.service.loading$, { initialValue: false });
  filter = toSignal(this.service.filter$, { initialValue: '' });

  ngOnInit() { this.service.load(); }
}

What this comparison reveals at the file level:

  • SignalTree — 3 source files (tree, ops, component) + 1 provider line in app.config.ts. Tree owns all reads; Ops owns all writes; entity normalization is built-in via entityMap().
  • NgRx SignalStore — 1 source file + component. signalStore() fuses state + reads + writes; providedIn: 'root' auto-registers; entity normalization is built-in via withEntities(); reactive async via rxMethod().
  • Classic NgRx — 5 source files (actions, reducer, selectors, effects, component) + 2 lines in app.config.ts (provideStore + provideEffects). The action/reducer/effect cycle is genuine boilerplate.
  • BehaviorSubject service — 1 source file + component. Smallest in source files, but you write entity normalization (the Map<string, Ticket>) by hand to match what the other three deliver out of the box.

Where the savings actually come from: against modern NgRx SignalStore the file-count win for SignalTree is small — both are typed-signal stores. Against Classic NgRx the savings are large because the action/reducer/effect cycle dissolves into direct method calls on a tree. Against BehaviorSubject services you don't save files but you gain typed paths, entity normalization, DevTools, and time travel.

Feature 2: Optimistic update with rollback

The user edits a ticket title. The UI shows the change immediately. If the server returns an error, the title rolls back to the previous value and the error surfaces to the component.

Capability checklist — every implementation must:
  • Capture the current title before the optimistic update (rollback target)
  • Update the entity's title in state immediately
  • Clear any prior error
  • Call api.update(id, { title }) and await the result
  • On success: keep the new state, error stays null
  • On failure: rollback to the captured title, set error to a string
  • Component reads the (possibly-rolled-back) title and the error

All four implementations extend the state shape from Feature 1 with error: string | null.

2.1 SignalTree

// ticket.ops.ts
@Injectable({ providedIn: 'root' })
export class TicketOps {
  private tree = inject(APP_TREE);
  private api = inject(TicketApi);

  async updateTitle(id: string, title: string) {
    const previous = this.tree.$.tickets.byId(id)();
    if (!previous) return;

    // Optimistic
    this.tree.$.tickets.updateOne(id, { title });
    this.tree.$.error.set(null);

    try {
      await firstValueFrom(this.api.update(id, { title }));
    } catch (err) {
      // Rollback
      this.tree.$.tickets.updateOne(id, { title: previous.title });
      this.tree.$.error.set((err as Error).message);
    }
  }
}

2.4 Plain BehaviorSubject service

async updateTitle(id: string, title: string) {
  const previous = this._byId.get(id);
  if (!previous) return;

  // Optimistic
  this._byId.set(id, { ...previous, title });
  this._ids$.next([...this._ids$.value]);  // re-emit to invalidate visible$
  this._error$.next(null);

  try {
    await firstValueFrom(this.api.update(id, { title }));
  } catch (err) {
    // Rollback
    this._byId.set(id, previous);
    this._ids$.next([...this._ids$.value]);
    this._error$.next((err as Error).message);
  }
}

What this comparison reveals: SignalTree, NgRx SignalStore, and BehaviorSubject all express optimistic+rollback as a plain async method that captures the previous value via direct lookup and rolls back in a catch block. Classic NgRx must round-trip through three actions (UpdateTitle, UpdateTitleSuccess, UpdateTitleFailure) and thread previousTitle through the action payload because the reducer is pure and cannot capture state at dispatch time. The BehaviorSubject version replicates SignalTree's mechanic but must manually re-emit _ids$ to invalidate the derived visible$ observable.

Feature 3: Reactive side effect on state change

When selectedUserId changes (and is non-null), fetch orders for that user. If a fetch is in flight when the id changes again, cancel it. This is SignalTree's REACT pillar in action.

Capability checklist — every implementation must:
  • Hold selectedUserId (string | null), orders (Order[]), ordersLoading (boolean)
  • React to changes in selectedUserId
  • If id is null: clear orders, do nothing else
  • If id is non-null: fetch orders for that id, set ordersLoading around the fetch
  • Cancel/discard any in-flight fetch when the id changes (no race conditions)
  • Component reads selected, orders, ordersLoading

3.1 SignalTree

Requires .with(effects()) on the tree to enable tree.effect(). 

// app.tree.ts — apply the effects enhancer
export function createAppTree() {
  return signalTree({
    selectedUserId: null as string | null,
    orders: [] as Order[],
    ordersLoading: false,
  }).with(effects())
    .derived(/* ... */);
}

// user.ops.ts
@Injectable({ providedIn: 'root' })
export class UserOps {
  private tree = inject(APP_TREE);
  private api = inject(OrderApi);

  constructor() {
    // REACT — runs whenever selectedUserId changes
    this.tree.effect((state) => {
      const userId = state.selectedUserId();

      if (!userId) {
        untracked(() => this.tree.$.orders.set([]));
        return;
      }

      const controller = new AbortController();
      untracked(() => {
        this.tree.$.ordersLoading.set(true);
        this.api.list(userId, { signal: controller.signal })
          .then((orders) => this.tree.$.orders.set(orders))
          .catch(() => {/* aborted or failed */})
          .finally(() => this.tree.$.ordersLoading.set(false));
      });

      return () => controller.abort();  // cleanup on next run
    });
  }

  selectUser(id: string | null) {
    this.tree.$.selectedUserId.set(id);
  }
}

3.4 Plain BehaviorSubject service

@Injectable({ providedIn: 'root' })
export class UserService implements OnDestroy {
  private _selectedUserId$ = new BehaviorSubject<string | null>(null);
  private _orders$ = new BehaviorSubject<Order[]>([]);
  private _ordersLoading$ = new BehaviorSubject<boolean>(false);
  private destroyed = new Subject<void>();

  selectedUserId$ = this._selectedUserId$.asObservable();
  orders$ = this._orders$.asObservable();
  ordersLoading$ = this._ordersLoading$.asObservable();

  constructor(private api: OrderApi) {
    // React to selectedUserId — switchMap cancels prior fetch
    this._selectedUserId$.pipe(
      switchMap((userId) => {
        if (!userId) {
          this._orders$.next([]);
          this._ordersLoading$.next(false);
          return EMPTY;
        }
        this._ordersLoading$.next(true);
        return this.api.list(userId).pipe(
          tap((orders) => this._orders$.next(orders)),
          finalize(() => this._ordersLoading$.next(false)),
          catchError(() => EMPTY)
        );
      }),
      takeUntil(this.destroyed)
    ).subscribe();
  }

  selectUser(id: string | null) {
    this._selectedUserId$.next(id);
  }

  ngOnDestroy() {
    this.destroyed.next();
    this.destroyed.complete();
  }
}

What this comparison reveals: all four libraries can cancel in-flight requests on state change — the difference is the mental model. SignalTree's tree.effect() looks like a synchronous callback over state with a cleanup return. NgRx SignalStore's rxMethod + withHooks wires a Signal source through a switchMap pipeline. Classic NgRx's effect listens for the action rather than the state change, but the cancellation semantics via switchMap are identical. The BehaviorSubject version must manage its own subscription lifetime via OnDestroy + takeUntil.

Feature 4: Async load with error state and retry

Extends Feature 1 with explicit error state and a retry() method. When the load fails, the previous data is preserved (not cleared), the error surfaces to the component, and the user can retry.

Capability checklist — every implementation must:
  • Hold error: string | null in state
  • On load(): clear error, set loading, call API
  • On success: items populated, loading false, error null
  • On failure: error captured, loading false, items preserved (not cleared)
  • Provide a retry() method that re-runs the load
  • Component reads items, loading, error; renders an error+retry UI

4.1 SignalTree

// ticket.ops.ts
async load() {
  this.tree.$.loading.set(true);
  this.tree.$.error.set(null);
  try {
    const tickets = await firstValueFrom(this.api.list());
    this.tree.$.tickets.setAll(tickets);
  } catch (err) {
    this.tree.$.error.set((err as Error).message);
    // Note: tickets entity collection is NOT cleared on failure
  } finally {
    this.tree.$.loading.set(false);
  }
}

retry() {
  return this.load();
}

4.4 Plain BehaviorSubject service

// ticket.service.ts (extends Feature 1)
private _error$ = new BehaviorSubject<string | null>(null);
error$ = this._error$.asObservable();

async load() {
  this._loading$.next(true);
  this._error$.next(null);
  try {
    const tickets = await firstValueFrom(this.api.list());
    this._byId.clear();
    tickets.forEach((t) => this._byId.set(t.id, t));
    this._ids$.next(tickets.map((t) => t.id));

    // Note: _byId / _ids$ are NOT cleared on failure — entities preserved
  } finally {

  }
}

  return this.load();
}

What this comparison reveals: the patterns converge at this level. SignalTree, SignalStore, and BehaviorSubject all use the same try / catch / finally shape; the only difference is where state lives. Classic NgRx's action/reducer/effect cycle gives you named failure semantics through the loadFailure action — which is more discoverable in DevTools but not functionally different. The "items preserved on failure" requirement is naturally satisfied everywhere because none of the failure paths clear the entity collection.

@ngrx/signals (signalStore)

The closest relative — both libraries give you typed signals over a state object. SignalTree replaces the withX composition model with a single root tree and plain Ops services.

Full implementation in both libraries: see Feature 1 above for a checklisted side-by-side.

Architectural difference: NgRx SignalStore keeps state, computed reads, and write methods in one store factory. SignalTree splits the same responsibilities across a shared tree (state + reads) and per-domain Ops services (writes). The split scales better when many features share state; the fused factory is fewer files for a single-feature store.

Full playbook: the deep migration guide with three strategies (big-bang, incremental, hybrid), a Phase 0 recipe, sequencing rules, and a verifier script lives at docs/skills/using-signaltree/reference/migration-from-ngrx-signals.md. It also ships inside @signaltree/core as an Agent Skill — see AI-assisted migration.

Plain BehaviorSubject services

The classic private _x$ = new BehaviorSubject<T>(initial) pattern becomes an Angular signal-backed service. You keep the mental model — services own the state — and gain typed paths, computed(), DevTools, and time travel for free.

Full implementation in both styles: see Feature 1 above for a checklisted side-by-side including manual entity normalization (which RxJS doesn't ship).

Quick swap for a simple feature

For a single self-contained service with no entities, the swap is mechanical — same file, same class, same registration:

Before — cart.service.ts

@Injectable({ providedIn: 'root' })
export class CartService {
  private _items$ = new BehaviorSubject<Item[]>([]);
  items$ = this._items$.asObservable();
  count$ = this.items$.pipe(map(xs => xs.length));

  add(item: Item) {
    this._items$.next([...this._items$.value, item]);
  }
}

After — cart.service.ts

@Injectable({ providedIn: 'root' })
export class CartService {
  private tree = signalTree({ items: [] as Item[] })
    .derived($ => ({
      count: computed(() => $.items().length)
    }));

  items = this.tree.$.items;   // Signal<Item[]>
  count = this.tree.$.count;   // Signal<number> — from .derived()

  add(item: Item) {
    this.tree.$.items.update(xs => [...xs, item]);
  }
}

For multi-domain apps: if you have many feature services and want a single shared state graph, SignalTree's recommended pattern is one shared AppTree (state + derived) plus per-domain Ops services for writes — see Feature 1 for that layout. Embedding signalTree inside an existing service (as shown here) is the simplest 1:1 swap and works just as well for single-feature services.

Plain Angular signals + service

If you already have signal() + computed() inside an @Injectable, you mostly just relocate the state into the shared tree. The wins are: one place to inspect everything in DevTools, time travel, persistence, and a single well-typed graph instead of N independent service-owned signals.

When you might not migrate: truly local UI state (collapsed/expanded, hover, transient form state) can stay as plain signal() in the component. The tree is for application state.

AI-assisted migration

SignalTree ships a vendor-neutral Agent Skill that lets Cursor, Claude Code, or any SKILL.md-aware AI assistant run these migrations end-to-end. After installing @signaltree/core the skill lives at node_modules/@signaltree/core/skills/using-signaltree/.

Install the skill once per repo

# Cursor
cp -r node_modules/@signaltree/core/skills/using-signaltree .cursor/skills/

# Claude Code
cp -r node_modules/@signaltree/core/skills/using-signaltree .claude/skills/

What the skill gives the agent

  • Mechanical concept maps for every source library listed above (the deepest coverage is currently @ngrx/signals; PRs welcome for the others).
  • Three migration strategies with explicit decision criteria — big-bang (one PR), incremental per-domain (one PR per store, with a Phase 0 foundation PR and a sequencing rule), and a hybrid legacy-facade for regulated rollback.
  • A drop-in verifier script (scripts/verify-signaltree-migration.sh in the SignalTree repo) that runs build + test + lint and asserts the source library is gone from application code and package.json. Package-manager and layout agnostic.
  • Anti-pattern catalog (root-injected Ops with eager side-effects, multiple competing trees, brown-field APP_TREE placement) so the agent doesn't repeat known traps.

Point your AI assistant at the skill, give it a target store to migrate, and it will follow the playbook and run the verifier itself. See Using SignalTree with AI Agents for harness-specific setup.
