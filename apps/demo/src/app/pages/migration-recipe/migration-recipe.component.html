Migrating to SignalTree

SignalTree is designed to replace the most common Angular state-management patterns without changing what the feature actually does. This page is now structured by state shape instead of by incremental feature deltas: one example for plain JSON object state, one for CRUD entity state, and one for reactive shared app state.

The examples were audited against the real v3 NgRx SignalStore app and the v3_ALT SignalTree migration branch. If another implementation needs supporting files, registrations, selectors, or component bindings to work, those supporting pieces are included in the snippet. The point is to show where SignalTree is genuinely simpler, not to let any library win by hiding code off-screen.

Example shapes:
  • Plain JSON Object State
  • CRUD Entity State
  • Reactive Shared App State
Library concept maps (quick reference):
  • @ngrx/signals (signalStore)
  • Classic NgRx (@ngrx/store + effects)
  • @ngrx/component-store
  • NGXS
  • Elf / Akita
  • @rx-angular/state
  • Plain BehaviorSubject services
  • Plain Angular signals + service
  • AI-assisted migration

Plain JSON Object State

Start simple: a cart service with one JSON object, one derived count, and one write method. This is the smallest realistic migration surface.

BehaviorSubject service

Files: cart.service.ts, component. 

@Injectable({ providedIn: 'root' })
export class CartService {
  private _items$ = new BehaviorSubject<Item[]>([]);
  items$ = this._items$.asObservable();
  count$ = this.items$.pipe(map((xs) => xs.length));

  add(item: Item) {
    this._items$.next([...this._items$.value, item]);
  }
}

@Component({ selector: 'app-cart', template: '...' })
export class CartComponent {
  service = inject(CartService);

  items = toSignal(this.service.items$, { initialValue: [] });
  count = toSignal(this.service.count$, { initialValue: 0 });
}

SignalTree service

Files: cart.service.ts, component. 

@Injectable({ providedIn: 'root' })
export class CartService {
  private tree = signalTree({ items: [] as Item[] })
    .derived(($) => ({
      count: computed(() => $.items().length)
    }));

  items = this.tree.$.items;
  count = this.tree.$.count;

  add(item: Item) {
    this.tree.$.items.update((xs) => [...xs, item]);
  }
}

@Component({ selector: 'app-cart', template: '...' })
export class CartComponent {
  service = inject(CartService);

  items = this.service.items;
  count = this.service.count;
}

What this shows: for plain object state, SignalTree can stay as mechanical as a BehaviorSubject service. The extra value is typed paths and computed reads, without forcing a whole app-wide architecture where one is not needed.

// derived/tickets.derived.ts
export function ticketsDerived($: AppTreeBase) {
  return {
    tickets: {
      visible: computed(() =>
        $.tickets.entities.all().filter((t) => t.title.includes($.tickets.filter()))
      ),
    },
  };
}

// tree/app-tree.ts
export function createAppTree() {
  return signalTree({ tickets: ticketsState() })
    .with(devTools({ treeName: 'AppTree' }))
    .with(batching())
    .with(memoization())
    .with(timeTravel())
    .derived(ticketsDerived);
}

// ops/tickets.ops.ts
@Injectable({ providedIn: 'root' })
export class TicketsOps {
  private readonly _$ = inject(APP_TREE).$.tickets;
  private readonly _api = inject(TicketApi);

  setFilter(filter: string) {
    this._$.filter.set(filter);
  }

  load$(): Observable<void> {
    this._$.loading.set(true);
    this._$.error.set(null);
    return this._api.list().pipe(
      tap((tickets) => this._$.entities.setAll(tickets)),
      tap(() => this._$.loading.set(false)),
      map(() => void 0),
      catchError((err) => {
        this._$.error.set(err.message);
        this._$.loading.set(false);
        return EMPTY;
      })
    );
  }

  async updateTitle(id: string, title: string): Promise<void> {
    const previous = this._$.entities.byId(id)();
    if (!previous) return;

    this._$.entities.updateOne(id, { title });
    this._$.error.set(null);

    try {
      await firstValueFrom(this._api.update(id, { title }));
    } catch (err) {
      this._$.entities.updateOne(id, { title: previous.title });
      this._$.error.set((err as Error).message);
    }
  }

  retry$(): Observable<void> {
    return this.load$();
  }
}

NgRx SignalStore

Files: tickets.store.ts, component. 

// tickets.store.ts
export const TicketStore = signalStore(
  { providedIn: 'root' },
  withDevtools('TicketStore'),
  withState({
    filter: '',
    loading: false,
    error: null as string | null,
  }),
  withEntities<Ticket>(),
  withComputed((store) => ({
    visible: computed(() =>
      store.entities().filter((t) => t.title.includes(store.filter()))
    ),
  })),
  withMethods((store, api = inject(TicketApi)) => ({
    setFilter(filter: string) {
      patchState(store, { filter });
    },

    load: rxMethod<void>(
      pipe(
        tap(() => patchState(store, { loading: true, error: null })),
        switchMap(() =>
          api.list().pipe(
            tap((tickets) =>
              patchState(store, setAllEntities(tickets), { loading: false })
            ),
            catchError((err) => {
              patchState(store, { loading: false, error: err.message });
              return EMPTY;
            })
          )
        )
      )
    ),

    async updateTitle(id: string, title: string) {
      const previous = store.entityMap()[id];
      if (!previous) return;

      patchState(
        store,
        updateEntity({ id, changes: { title } }, { selectId: (t) => t.id }),
        { error: null }
      );

      try {
        await firstValueFrom(api.update(id, { title }));
      } catch (err) {
        patchState(
          store,
          updateEntity({ id, changes: { title: previous.title } }, { selectId: (t) => t.id }),
          { error: (err as Error).message }
        );
      }
    },

    retry() {
      this.load();
    },
  }))
);

// tickets.reducer.ts
export interface TicketsState extends EntityState<Ticket> {
  filter: string;
  loading: boolean;
  error: string | null;
}

export const adapter = createEntityAdapter<Ticket>();
const initialState: TicketsState = adapter.getInitialState({
  filter: '',
  loading: false,
  error: null,
});

export const ticketsReducer = createReducer(
  initialState,
  on(TicketsActions.load, (s) => ({ ...s, loading: true, error: null })),
  on(TicketsActions.loadSuccess, (s, { tickets }) =>
    adapter.setAll(tickets, { ...s, loading: false })
  ),
  on(TicketsActions.loadFailure, (s, { error }) => ({ ...s, loading: false, error })),
  on(TicketsActions.setFilter, (s, { filter }) => ({ ...s, filter })),
  on(TicketsActions.updateTitle, (s, { id, title }) =>
    adapter.updateOne({ id, changes: { title } }, { ...s, error: null })
  ),
  on(TicketsActions.updateTitleFailure, (s, { id, previousTitle, error }) =>
    adapter.updateOne({ id, changes: { title: previousTitle } }, { ...s, error })
  )
);

// tickets.selectors.ts
const selectTicketsState = createFeatureSelector<TicketsState>('tickets');
const { selectAll } = adapter.getSelectors(selectTicketsState);
export const selectVisible = createSelector(
  selectAll,
  createSelector(selectTicketsState, (s) => s.filter),
  (tickets, filter) => tickets.filter((t) => t.title.includes(filter))
);

// tickets.effects.ts
@Injectable()
export class TicketsEffects {
  loadTickets$ = createEffect(() =>
    this.actions$.pipe(
      ofType(TicketsActions.load),
      switchMap(() =>
        this.api.list().pipe(
          map((tickets) => TicketsActions.loadSuccess({ tickets })),
          catchError((err) => of(TicketsActions.loadFailure({ error: err.message })))
        )
      )
    )
  );

  updateTitle$ = createEffect(() =>
    this.actions$.pipe(
      ofType(TicketsActions.updateTitle),
      mergeMap(({ id, title, previousTitle }) =>
        this.api.update(id, { title }).pipe(
          map(() => TicketsActions.updateTitleSuccess({ id })),
          catchError((err) =>
            of(TicketsActions.updateTitleFailure({ id, previousTitle, error: err.message }))
          )
        )
      )
    )
  );

  constructor(private actions$: Actions, private api: TicketApi) {}
}

  loading = toSignal(this.store.select((state) => state.tickets.loading), { initialValue: false });
  error = toSignal(this.store.select((state) => state.tickets.error), { initialValue: null });

    const previousTitle = this.store.selectSignal((state) => state.tickets.entities[id])()?.title;
    if (!previousTitle) return;

  }

  retry() {
    this.store.dispatch(TicketsActions.load());
  }
}

BehaviorSubject service

Files: ticket.service.ts, component. 

// ticket.service.ts
@Injectable({ providedIn: 'root' })
export class TicketService {
  private _byId = new Map<string, Ticket>();
  private _ids$ = new BehaviorSubject<string[]>([]);
  private _filter$ = new BehaviorSubject<string>('');
  private _loading$ = new BehaviorSubject<boolean>(false);
  private _error$ = new BehaviorSubject<string | null>(null);

  filter$ = this._filter$.asObservable();
  loading$ = this._loading$.asObservable();
  error$ = this._error$.asObservable();

  all$ = this._ids$.pipe(
    map((ids) => ids.map((id) => this._byId.get(id)!).filter(Boolean))
  );

  visible$ = combineLatest([this.all$, this.filter$]).pipe(
    map(([tickets, filter]) => tickets.filter((t) => t.title.includes(filter)))
  );

  constructor(private api: TicketApi) {}

  setFilter(filter: string) {
    this._filter$.next(filter);
  }

  async load() {
    this._loading$.next(true);
    this._error$.next(null);
    try {
      const tickets = await firstValueFrom(this.api.list());
      this._byId.clear();
      tickets.forEach((t) => this._byId.set(t.id, t));
      this._ids$.next(tickets.map((t) => t.id));
    } catch (err) {
      this._error$.next((err as Error).message);
    } finally {
      this._loading$.next(false);
    }
  }

  async updateTitle(id: string, title: string) {
    const previous = this._byId.get(id);
    if (!previous) return;

    this._byId.set(id, { ...previous, title });
    this._ids$.next([...this._ids$.value]);
    this._error$.next(null);

    try {
      await firstValueFrom(this.api.update(id, { title }));
    } catch (err) {
      this._byId.set(id, previous);
      this._ids$.next([...this._ids$.value]);
      this._error$.next((err as Error).message);
    }
  }

  retry() {
    return this.load();
  }
}

Where SignalTree wins honestly: the read/write density is close to NgRx SignalStore on a single domain. The larger savings come from configuring cross-cutting concerns once on the shared tree and then adding state slices, derived tiers, and domain ops as the app grows.

Reactive Shared App State

This example shows one shared state change causing another domain to react: when selectedUserId changes, fetch orders for that user and cancel any in-flight request.

Capability checklist:
  • Shared selectedUserId state
  • Orders list plus loading state
  • Cancel prior request when selection changes
  • Clear orders on null selection
  • Component reads selectedUserId, orders, and ordersLoading

SignalTree

// state/selection.state.ts
export function selectionState() {
  return { selectedUserId: null as string | null };
}

// state/orders.state.ts
export function ordersState() {
  return { list: [] as Order[], loading: false };
}

// tree/app-tree.ts
export function createAppTree() {
  return signalTree({
    selection: selectionState(),
    orders: ordersState(),
  })
    .with(effects())
    .with(devTools({ treeName: 'AppTree' }))
    .with(batching());
}

// ops/orders.ops.ts
@Injectable({ providedIn: 'root' })
export class OrdersOps {
  private readonly _tree = inject(APP_TREE);
  private readonly _$ = this._tree.$;
  private readonly _api = inject(OrderApi);

  constructor() {
    this._tree.effect((state) => {
      const userId = state.selection.selectedUserId();

      if (!userId) {
        untracked(() => this._$.orders.list.set([]));
        return;
      }

      const controller = new AbortController();
      untracked(() => {
        this._$.orders.loading.set(true);
        this._api.list(userId, { signal: controller.signal })
          .then((orders) => this._$.orders.list.set(orders))
          .finally(() => this._$.orders.loading.set(false));
      });

      return () => controller.abort();
    });
  }

  selectUser(id: string | null) {
    this._$.selection.selectedUserId.set(id);
  }
}

// users.component.ts
@Component({ selector: 'app-users', template: '...' })
export class UsersComponent {
  private tree = inject(APP_TREE);
  ops = inject(OrdersOps);

  selectedUserId = this.tree.$.selection.selectedUserId;
  orders = this.tree.$.orders.list;
  ordersLoading = this.tree.$.orders.loading;
}

BehaviorSubject service

Files: user.service.ts, component. 

// user.service.ts
@Injectable({ providedIn: 'root' })
export class UserService implements OnDestroy {
  private _selectedUserId$ = new BehaviorSubject<string | null>(null);
  private _orders$ = new BehaviorSubject<Order[]>([]);
  private _ordersLoading$ = new BehaviorSubject<boolean>(false);
  private destroyed = new Subject<void>();

  selectedUserId$ = this._selectedUserId$.asObservable();
  orders$ = this._orders$.asObservable();
  ordersLoading$ = this._ordersLoading$.asObservable();

  constructor(private api: OrderApi) {
    this._selectedUserId$.pipe(
      switchMap((userId) => {
        if (!userId) {
          this._orders$.next([]);
          this._ordersLoading$.next(false);
          return EMPTY;
        }
        this._ordersLoading$.next(true);
        return this.api.list(userId).pipe(
          tap((orders) => this._orders$.next(orders)),
          finalize(() => this._ordersLoading$.next(false)),
          catchError(() => EMPTY)
        );
      }),
      takeUntil(this.destroyed)
    ).subscribe();
  }

  selectUser(id: string | null) {
    this._selectedUserId$.next(id);
  }

  ngOnDestroy() {
    this.destroyed.next();
    this.destroyed.complete();
  }
}

What this shows: all four approaches can model reactive cancellation. SignalTree's distinction is that the reaction is attached to the shared state graph itself, so cross-domain reads stay local to the tree instead of being re-wired through store-local hooks or action cycles.

@ngrx/signals (signalStore)

The closest relative — both libraries give you typed signals over a state object. SignalTree replaces the withX composition model with a single root tree and plain Ops services.

Full implementation in both libraries: see CRUD Entity State above for the file-complete side-by-side.

Concept map

withReduxDevtools()devTools() enhancer

Architectural difference: NgRx SignalStore keeps state, computed reads, and write methods in one store factory. SignalTree splits the same responsibilities across a shared tree (state + reads) and per-domain Ops services (writes). The practical payoff is app-wide configuration and cross-domain reads live in one graph, while feature writes stay local to their domain service. The fused factory is fewer files for a single-feature store.

Full playbook: the deep migration guide with three strategies (big-bang, incremental, hybrid), a Phase 0 recipe, sequencing rules, and a verifier script lives at docs/skills/using-signaltree/reference/migration-from-ngrx-signals.md. It also ships inside @signaltree/core as an Agent Skill — see AI-assisted migration.

Classic NgRx (@ngrx/store + @ngrx/effects)

The actions / reducers / selectors / effects model collapses into a single tree plus async methods on an Ops service. Most "type the action, dispatch the action, handle the action" plumbing disappears.

Full implementation in both libraries: see CRUD Entity State above for the file-complete side-by-side.

Strategy: migrate one feature module at a time. Build the AppTree + a feature Ops service alongside the existing reducer, switch the components, then delete the actions / reducer / effects / selectors files in the same PR.

@ngrx/component-store

Component Store is closest in shape — local state, updaters, effects. Map each ComponentStore to an Ops service bound to a tree slice (or to a small per-component signalTree() if it really is component-scoped).

Plain BehaviorSubject services

The classic private _x$ = new BehaviorSubject<T>(initial) pattern becomes an Angular signal-backed service. You keep the mental model — services own the state — and gain typed paths, computed(), DevTools, and time travel for free.

Full implementation in both styles: see CRUD Entity State above for the file-complete side-by-side including manual entity normalization.

Quick swap for a simple feature

For a single self-contained service with no entities, the swap is mechanical — same file, same class, same registration:

Before — cart.service.ts

@Injectable({ providedIn: 'root' })
export class CartService {
  private _items$ = new BehaviorSubject<Item[]>([]);
  items$ = this._items$.asObservable();
  count$ = this.items$.pipe(map(xs => xs.length));

  add(item: Item) {
    this._items$.next([...this._items$.value, item]);
  }
}

After — cart.service.ts

@Injectable({ providedIn: 'root' })
export class CartService {
  private tree = signalTree({ items: [] as Item[] })
    .derived($ => ({
      count: computed(() => $.items().length)
    }));

  items = this.tree.$.items;   // Signal<Item[]>
  count = this.tree.$.count;   // Signal<number> — from .derived()

  add(item: Item) {
    this.tree.$.items.update(xs => [...xs, item]);
  }
}

For multi-domain apps: if you have many feature services and want a single shared state graph, SignalTree's recommended pattern is one shared AppTree (state + derived) plus per-domain Ops services for writes — see Shared App State for that layout. Embedding signalTree inside an existing service (as shown here) is the simplest 1:1 swap and works just as well for single-feature services.

Plain Angular signals + service

If you already have signal() + computed() inside an @Injectable, you mostly just relocate the state into the shared tree. The wins are: one place to inspect everything in DevTools, time travel, persistence, and a single well-typed graph instead of N independent service-owned signals.

When you might not migrate: truly local UI state (collapsed/expanded, hover, transient form state) can stay as plain signal() in the component. The tree is for application state.

AI-assisted migration

SignalTree ships a vendor-neutral Agent Skill that lets Cursor, Claude Code, or any SKILL.md-aware AI assistant run these migrations end-to-end. After installing @signaltree/core the skill lives at node_modules/@signaltree/core/skills/using-signaltree/.

Install the skill once per repo

# Cursor
cp -r node_modules/@signaltree/core/skills/using-signaltree .cursor/skills/

# Claude Code
cp -r node_modules/@signaltree/core/skills/using-signaltree .claude/skills/

What the skill gives the agent

  • Mechanical concept maps for every source library listed above (the deepest coverage is currently @ngrx/signals; PRs welcome for the others).
  • Three migration strategies with explicit decision criteria — big-bang (one PR), incremental per-domain (one PR per store, with a Phase 0 foundation PR and a sequencing rule), and a hybrid legacy-facade for regulated rollback.
  • A drop-in verifier script (scripts/verify-signaltree-migration.sh in the SignalTree repo) that runs build + test + lint and asserts the source library is gone from application code and package.json. Package-manager and layout agnostic.
  • Anti-pattern catalog (root-injected Ops with eager side-effects, multiple competing trees, brown-field APP_TREE placement) so the agent doesn't repeat known traps.

Point your AI assistant at the skill, give it a target store to migrate, and it will follow the playbook and run the verifier itself. See Using SignalTree with AI Agents for harness-specific setup.
