SignalTree: Reactive JSON

JSON branches, reactive leaves. No actions. No reducers. No selectors.

🎯 Why Consolidation Matters

Stop choosing between features and bundle size. Traditional state management forces you to pick: install separate packages for every feature (NgRx: @ngrx/store, @ngrx/effects, @ngrx/entity, @ngrx/router-store, @ngrx/component-store...) or settle for limited functionality.

📦 46% Smaller Dependencies

Snapshot from one production app: NgRx (4 packages) ~45-50KB gzipped → SignalTree (1 package) ~27KB gzipped. Tree-shakable architecture means you only ship what you use. Full core with all enhancers included. YMMV.

🔧 Zero Version Conflicts

NgRx's separate packages create peer dependency hell. Need @ngrx/store@16.x with @ngrx/effects@16.x and @ngrx/entity@16.x? Hope they're compatible. SignalTree: one package, one version, zero conflicts.

⚡ Optimized Together

Separate packages can't share optimizations. SignalTree's batching and lazy evaluation work across all features because they're designed as a unified system.

One production-app snapshot: NgRx (4 packages) = ~45-50KB gzipped. SignalTree core (includes all enhancers) = ~27KB gzipped. 46% bundle reduction, 76% less app code, 75% fewer packages.

*Snapshot from one production Angular mobile app's NgRx Signal Store → SignalTree migration. Both codebases have continued to evolve; re-measuring today yields a 60–70% reduction depending on scope. The directional finding is reproducible — the exact percentages are not. YMMV — results depend on app complexity, prior architecture, and how heavily the original code leaned on custom withX helpers. The biggest driver of savings is cross-cutting concerns (devtools, error banners, telemetry) moving from per-store composition to tree-level enhancers.

The 3-Pillar Pattern

The recommended multi-domain architecture follows one rule — and once you know it, the pattern is obvious everywhere:

READ

.derived()

All computed state lives on the tree — never inside Ops services or component classes. Components read directly from tree.$. 

// app.tree.ts
signalTree({ tickets: entityMap<Ticket>(), filter: '' })
  .derived($ => ({
    tickets: {
      visible: computed(() =>
        $.tickets.all().filter(t => t.title.includes($.filter()))
      )
    }
  }));

// component — one inject, all reads available
tree.$.tickets.visible()
WRITE

Ops services

Mutations and async live in thin @Injectable services. No computed properties belong here. 

@Injectable({ providedIn: 'root' })
export class TicketOps {
  private tree = inject(APP_TREE);
  private api  = inject(TicketApi);

  async load() {
    const data = await firstValueFrom(this.api.list());
    this.tree.$.tickets.setAll(data); // write only
  }
}
REACT

tree.effect()

State changes are the events. No actions, no dispatch, no effects registry — just react to state. 

// Registered once in a root service
tree.effect(state => {
  const filters = state.tickets.filters;
  // re-runs whenever filters change
  untracked(() => ticketOps.load(filters));
});
