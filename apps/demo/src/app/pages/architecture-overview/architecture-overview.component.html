SignalTree: Reactive JSON

JSON branches, reactive leaves. No actions. No reducers. No selectors.

🎯 Why Consolidation Matters

Model state as data, not framework ceremony. SignalTree ships a single package where entities, batching, DevTools, time travel, and persistence are enhancers you opt into — unused ones tree-shake away.

📦 Less State Code

Snapshot from one production Angular app's migration off @ngrx/signals (SignalStore): its shipped state-code bundle went ~50KB → ~27KB gzipped, driven mostly by writing ~76% fewer lines of state code. This is the migrated app's state bundle, not the SignalTree library size (core is ~8.5KB gzipped). YMMV.

🔧 One Package, One Version

SignalTree is a single versioned package: no cross-package peer ranges to reconcile. (Classic NgRx splits @ngrx/store, @ngrx/effects, @ngrx/entity… across separate installs, but that sprawl is not what the measurement above came from.)

⚡ Optimized Together

All enhancers are designed as a unified system, so SignalTree's batching and lazy evaluation work across every feature without cross-package integration seams.

One production-app snapshot (migration off @ngrx/signals): the app's shipped state-code bundle went ~50KB → ~27KB gzipped. ~46% state-bundle reduction, ~76% less app code. These are one app's numbers, not the library's — the SignalTree core library is ~8.5KB gzipped.

*Snapshot from one production Angular mobile app's NgRx Signal Store (@ngrx/signals) → SignalTree migration. Both codebases have continued to evolve; re-measuring today yields a 60–70% reduction depending on scope. The directional finding is reproducible — the exact percentages are not. YMMV — results depend on app complexity, prior architecture, and how heavily the original code leaned on custom withX helpers. The biggest driver of savings is cross-cutting concerns (devtools, error banners, telemetry) moving from per-store composition to tree-level enhancers.

The 3-Pillar Pattern

The recommended multi-domain architecture follows one rule — and once you know it, the pattern is obvious everywhere:

READ

.derived()

All computed state lives on the tree — never inside Ops services or component classes. Components read directly from tree.$.

WRITE

Ops services

Mutations and async live in thin @Injectable services. No computed properties belong here.

REACT

tree.effect()

State changes are the events. No actions, no dispatch, no effects registry — just react to state.
