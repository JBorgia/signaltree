🚀 What's New

🎯 v12.1.0 — idle()/settled() status predicates + persist.maxScopes (July 2026)

Two gaps closed from external audits: a status() guard predicate that actually covers the Error state, and storage garbage collection for high-cardinality scoped loaders.

✨ New

  • status().idle() / status().settled() — the standard composite predicates. idle() is !loading() && !loaded(), true for both NotLoaded and Error — guards/resolvers should use idle(), not notLoaded(): a notLoaded()-gated fetch silently never retries after an error, because the collection has moved to the distinct Error state. settled() is loaded() || hasError() ("done, stop the spinner"); idle() and settled() deliberately overlap while errored.
  • persist.maxScopes — persisted-scope garbage collection for scoped cache-aware collections (entityMap({ load: loader(fn, { persist }) })). Each scope persists under its own storage key, so high-cardinality scopes (tenants, customers, searches) previously accumulated entries forever. With maxScopes set, the loader keeps a touch-ordered index and evicts the least-recently-written scopes beyond the cap on every write-through. Unset = previous behavior (no GC). 
// idle() vs notLoaded() after an error
store.$.orgStatus.setError(err);
store.$.orgStatus.notLoaded();  // false — WRONG guard, never retries
store.$.orgStatus.idle();       // true  — RIGHT guard, covers NotLoaded + Error

// persist.maxScopes: bounded storage for scoped loaders
customers: entityMap<Customer, string, { tenantId: string }>({
  selectId: (c) => c.id,
  load: loader(({ tenantId }) => api.getCustomers$(tenantId), {
    persist: { adapter, key: 'customers', maxScopes: 3 },
  }),
}),

Try both live: the marker zoo's idle()/settled() panel, and the entityMap showcase's scoped maxScopes eviction panel.

📡 v11.2.0 — entityMap() cache-aware loading + invalidateTag() (July 2026)

A cache-aware, self-loading entityMap: the marker family gains a collection primitive that owns its own fetch, freshness, and invalidation — no more hand-wiring asyncSource + entityMap + manual staleness bookkeeping for the common "load a list, cache it, invalidate on demand" shape.

✨ New

  • entityMap<E, K>(config) with a load option — the full entityMap surface (all CRUD + query signals) now composes with a loader, load status, and cache semantics directly. .load() is guarded: a no-op while fresh or already in-flight, so concurrent callers coalesce onto one fetch. .refresh() forces a reload; .invalidate() marks the collection stale without fetching.
  • staleTime — freshness window (ms or a duration string like '30m') that .load() respects before refetching. swr (stale-while-revalidate) keeps loaded() true during background revalidation instead of flipping back to loading.
  • tags + invalidateTag(tree, tag) — register a collection under one or more tags, then invalidate every collection carrying that tag from anywhere with a single call. The push-invalidation seam for SSE/SignalR: an incoming "plants changed" event becomes invalidateTag(tree, 'plants'), no manual fan-out to each consumer.
  • Optional offline-first persistence — persist: { adapter, key, hydrateThenRevalidate } seeds rows instantly from a storage adapter on materialize, then revalidates via load() in the background. 
const tree = signalTree({
  plants: entityMap<PlantDto, string>({
    load: () => plantApi.list$(region),
    selectId: (p) => p.url,
    staleTime: '30m',     // skip refetch while fresh
    tags: ['plants'],     // invalidateTag(tree, 'plants')
  }),
});

tree.$.plants.all();        // PlantDto[] (full entityMap surface)
tree.$.plants.loading();    // Signal<boolean>
await tree.$.plants.load(); // guarded — no-op while fresh / in-flight
invalidateTag(tree, 'plants'); // marks every 'plants'-tagged collection stale

💥 v11.0.0 — Deprecations removed; tree-shakeable by default; linked() (June 2026)

The cleanup release the v10.3 deprecations pointed at, plus a real bundle-floor reduction and one new primitive.

⚠️ Breaking Changes

  • Deprecated is-prefix predicates removed. Use the v10.3 canonical bare names: status → loading / loaded / hasError / notLoaded (was isLoading / isLoaded / isError / isNotLoaded), and entityMap().empty (was isEmpty). All mechanical — TypeScript flags every call site.
  • Root tree.state accessor removed. Read (and write) through tree.$ everywhere — state was always the same reference, now retired.
  • security and lazy signals are opt-in. Wrap security config with security() from @signaltree/core/security, and inject lazy: lazy() from @signaltree/core/lazy to restore auto-lazy behavior. Both were statically reachable before and shipped in every bundle; they now tree-shake out when unused. Reads/writes are functionally identical.

✨ New

  • linked(...) — derived-but-writable signal. Wraps Angular's native linkedSignal: a value computed from a source that you can also write directly, re-deriving when the source changes (e.g. a "sticky selection"). Composes with serialization / persistence / snapshot. 
    // simple form
tree.derived($ => ({ doubled: linked(() => $.count() * 2) }));

// source + computation form (writable, re-derives on source change)
tree.derived($ => ({
  selected: linked({
    source: () => $.options(),
    computation: (opts, prev) => prev?.value ?? opts[0],
  }),
}));

tree.$.selected.set(next); // type-checks — it's a real WritableSignal
  • defineStore(factory, config?) — wraps a signalTree(...) factory in an injectable Angular service (inject(MyStore) resolves to the real tree; destroy() is tied to the host injector via DestroyRef). Tree-shakes out when unused.

📦 Bundle

Injecting the security + lazy/memory machinery drops the bare-tree gzip floor ~29% (7.5KB → ~5.3KB; ~8.1KB with entityMap in use), and devTools() now fully strips from production builds. Same release also corrected an old, false "smaller than NgRx SignalStore" claim — SignalStore is ~2.3KB and SignalTree is larger; the honest framing is capability-per-KB and zero runtime dependencies, not raw size.

🎯 v10.3.0 — Marker accessor shape UNIFIED (bare-name predicates) (May 2026)

A real DX bug the v10.2 benchmark surfaced: SignalTree's own markers named their predicate accessors inconsistently — status() used is-prefix (.isLoading()), entityMap had .isEmpty(), while form/asyncSource/asyncQuery already used bare names. v10.3 makes bare names canonical everywhere, matching FormControl.dirty / .valid and Angular signal conventions. 

tree.$.load.loading();   // was .isLoading()
tree.$.load.loaded();    // was .isLoaded()
tree.$.load.hasError();  // was .isError()
tree.$.users.empty();    // was .isEmpty()

No breaking changes in v10.3 — the is-prefix names became @deprecated aliases returning the same Signal instance (no double allocation), and were removed in v11.0. The deeper takeaway: the AI-codegen benchmark didn't just measure AI accuracy, it surfaced a genuine DX bug — AI-friendly and human-friendly turned out to be the same thing.

✨ v9.5.0 — Async markers: asyncSource and asyncQuery (May 2026)

The SignalTree-native answer to async state. Two new markers in the same family as entityMap, status, stored, and form: attach at any tree path, auto-derive data/loading/error signals, no manual tap()/setLoading() wiring.

✨ New

  • asyncSource<T>(config) — load-and-expose. Auto-loads on materialization. Exposes data, loading, error, plus refresh / set / update / reset. Accepts Observable or Promise loaders.
  • asyncQuery<TInput, TResult>(config) — input-driven debounced query. Writable input signal drives a debounce → filter → distinctUntilChanged → switchMap pipeline. Race conditions, cancellation, dedup all handled.
  • Live demo: /async

⚠️ v6.0.0 — Breaking Changes (summary)

SignalTree v6 tightens types and makes enhancers opt-in. Important user-facing changes (short summary):

  • Enhancer factories renamed: Short factories are now primary (e.g. persistence()); legacy withX() names remain as deprecated aliases.
  • Methods are added by enhancers: The base tree no longer declares all methods — apply an enhancer to gain its methods (e.g. time-travel, batching). This makes the type system reflect runtime behavior.
  • Polymorphic enhancer pattern: Enhancers are now polymorphic in the tree state type so TypeScript correctly infers updated APIs when you call .with().
  • Marker-based entity API: Use entityMap() in your initial state (marker) and access reactive CRUD via store.$.myEntities.*.
  • Typing & API cleanup: Removed index-signature workarounds and improved inference—prefer annotating initial values with per-field assertions (see Preferred Typing docs).

🔥 v5.0.0 (December 2025) - Major Release

⚠️ Breaking Changes

  • Middleware System Removed: The entire middleware system has been deleted. Use tap() and intercept() hooks on entity signals instead for lifecycle observation.
  • subscribe() throws outside injection context: Calling tree.subscribe() outside Angular's injection context now throws an Error instead of silently failing. Enhancers use polling fallback when needed.

✨ New Features

  • PathNotifier System: Internal notification system for path-based mutations. Enables entity hooks without global state.
  • Entity Hooks - tap(): Observe entity mutations (add/update/remove) after they occur. 
    store.$.users.tap({
  onAdd: (user, id) => console.log('Added:', user),
  onUpdate: (id, changes) => console.log('Updated:', id),
  onRemove: (id, user) => console.log('Removed:', user),
});
  • Entity Hooks - intercept(): Block or transform mutations before they happen. Perfect for validation! 
    store.$.users.intercept({
  onAdd: (user, ctx) => {
    if (!user.email.includes('@')) {
      ctx.block('Invalid email');
    }
  },
  onUpdate: (id, changes, ctx) => {
    ctx.transform({ ...changes, updatedAt: new Date() });
  },
});
  • Marker-Based Entity System: Type-safe EntityMapMarker<T, ID> with reactive CRUD operations via store.$.field.method()
  • Enhanced Types: Recursive inference up to 20+ nesting levels with improved IntelliSense
  • Angular 20.3 Support: Full compatibility with signal forms using connect() and toWritableSignal()
  • Tree-Shakeable Audit: Import audit tracking separately via @signaltree/ng-forms/audit

🚀 Performance Improvements

  • 2-5x faster bulk operations with diff-based updates in enterprise enhancer
  • 40.5% bundle size reduction with consolidated architecture and shared dependencies
  • Zero polling in Angular: subscribe() uses Angular effects when available, falling back to polling only in tests

*The speed and bundle figures above were measured on one app's workload at the time of the v5 release; your results will vary. See the current benchmarks for up-to-date, reproducible numbers.
