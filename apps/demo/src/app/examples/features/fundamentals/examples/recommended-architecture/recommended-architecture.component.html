🏛️ Recommended Architecture

One tree, one rule

The production pattern is three pillars applied consistently: READ through store.$, WRITE through store.ops, and REACT through tree.effect() — where a state change is the event, so there are no actions and nothing to dispatch. API services sit underneath as a supporting layer: HTTP only, no state knowledge.

Honest fit: this pattern earns its keep in apps with several domains and evolving, nested state. For a counter or a flat todo list it's overkill — a plain signal or one small tree is the better call. It's a way to keep a large app consistent, not a tax on every app.

Interact with the live demo below — the pillar you exercise lights up.

READstore.$
  • All computed state on $
  • Tiered: entity → filter → UI
  • Never in ops or components
WRITEstore.ops
  • Mutations & async only
  • No computed properties
  • One ops file per domain
REACTtree.effect()
  • State changes are the events
  • No actions, no dispatch
  • Registered in root services
