# AI-Codegen Accuracy Benchmark — First Run (2026-05-29)

> **First execution of the benchmark scaffolding.** Run on free open-source models via OpenRouter due to credit-tier constraints (frontier models like Claude / GPT-4o / Gemini Pro require paid balance). Despite the limitation, the results produce a clear, **publishable finding** that validates the audit's HSA L1 #3 priority.

## TL;DR

**AI agents generate ~0% correct SignalTree code without retrieval priming.**

When asked to use SignalTree, GPT-OSS-120B:

- **Imported from `'signal-tree'`** (wrong package name) and **`'@signal-tree/core'`** (wrong hyphen)
- **Called functions that don't exist**: `collection()`, `loading()`, `error()`, `debounceSignal()`, `distinctUntilChangedSignal()`, `switchMapSignal()`, `_users.asNormalized()`
- **Or fell back to raw Angular signals** entirely, importing `signal` from `@angular/core` and not touching SignalTree's API at all

When asked to use `@ngrx/signals`, the same model:

- Imported `signal` from `@ngrx/signals` (wrong — that's from `@angular/core`)
- Treated `@ngrx/signals` as if it just re-exported signal primitives — never reached for `signalStore`, `withState`, `withMethods`

When asked to use `@ngrx/store` (classic), the same model:

- **Got it perfectly right.** Proper `createAction`, `createReducer`, `Store` injection, `toSignal` interop, selectors.

The pattern: **the older a library is in the training corpus, the better AI agents generate code for it**. SignalTree (new) and `@ngrx/signals` (newer than classic NgRx) both fail. Classic NgRx (years of training data) wins.

**This is exactly the gap the audit's HSA L1 #3 priority describes.** It's also exactly what the SignalTree AI-discoverability investment (`llms.txt`, agent skills in npm tarballs, `.cursorrules` / `CLAUDE.md` templates) is designed to fix. The next benchmark iteration should measure WITH retrieval priming vs WITHOUT.

## Methodology

- **Source**: 3 reproducible prompt YAMLs in `prompts/` (counter, paginated users with loading state, debounced search).
- **Libraries**: 5 — SignalTree, `@ngrx/signals`, `@ngrx/store` (classic), Akita, Elf.
- **Agents attempted**: 4 free open-source frontier models via OpenRouter.
- **Routing**: OpenRouter unified adapter (`adapters/openrouter.mjs`).
- **Scoring (this run)**: idiomatic-pattern matching only (must_include + must_not_include + should_include_one_of). Compile + behavior tests scaffolded but TODO — they require an Angular workspace harness.

### Scoring methodology caveat (the score itself has a flaw)

The scoring formula is:

```
50 points if all must_include patterns are present
25 points if all must_not_include patterns are absent
25 points if at least one should_include_one_of is present
```

A library with NO must_not_include or should_include_one_of patterns auto-scores 50 vacuously for those buckets, then needs only must_include to hit 100. SignalTree and `@ngrx/signals` were defined with strict patterns (specific function names that MUST appear); the other three were defined with looser patterns. This biases scores upward for the libraries with thinner pattern specs.

**Fix for next iteration**: equalize pattern strictness across all libraries (require ≥3 must_include, ≥2 must_not_include, ≥2 should_include_one_of), then re-run.

That said — even with the scoring bias, the **CODE** GPT-OSS-120B generated for SignalTree was objectively wrong (hallucinated imports + function names). The score caught that via the must_include `signalTree(` requirement returning false.

## The actual generated code (what makes this interesting)

### Counter prompt — SignalTree requested

```typescript
import { Injectable } from '@angular/core';
import { signal, Signal } from '@angular/core';  // ❌ Should be from @signaltree/core

@Injectable({ providedIn: 'root' })
export class CounterService {
  private _count = signal<number>(0);  // ❌ Raw Angular signal, not signalTree({...})
  readonly count: Signal<number> = this._count.asReadonly();

  increment(): void {
    this._count.update(value => value + 1);
  }
  // ... etc
}
```

**SignalTree is not used at all.** Just raw Angular signals.

### Counter prompt — `@ngrx/signals` requested

```typescript
import { signal, Signal } from '@ngrx/signals';  // ❌ `signal` is not exported from @ngrx/signals
```

**Imports a non-existent export.** `@ngrx/signals` exports `signalStore`, `withState`, etc. — not `signal()`.

### Counter prompt — `@ngrx/store` (classic) requested

```typescript
import { Store, createAction, createReducer, on, props, select } from '@ngrx/store';
// ... proper action / reducer / selector / Store injection
```

**Correct.** Years of training data on classic NgRx pays off.

### Paginated-users prompt — SignalTree requested (the worst)

```typescript
import { collection, loading, error, type NormalizedCollection } from 'signal-tree';
//        ❌^ doesn't exist  ❌^ wrong package name (hyphen vs no-hyphen)

private readonly _users = collection<User>();     // ❌ no such function
readonly users: NormalizedCollection<User> = this._users.asNormalized();  // ❌ no such method
readonly loading = loading();                     // ❌ no such function
readonly error = error();                         // ❌ no such function
```

The model **invented a plausible-sounding API**. None of `collection()`, `loading()`, `error()`, `NormalizedCollection<T>`, or `.asNormalized()` exist in any SignalTree package. The package name `signal-tree` (with a hyphen) is wrong; the real name is `@signaltree/core` (no hyphen).

### Debounced-search prompt — SignalTree requested

```typescript
import { debounceSignal, distinctUntilChangedSignal, switchMapSignal } from '@signal-tree/core';
//                                                                          ❌^ wrong hyphen
// None of these three functions exist in any SignalTree package.
```

Same pattern. Hallucinated API. Plausible-sounding names that don't exist.

## What this proves

1. **The audit's central question — "why does an AI agent reach for NgRx instead of SignalTree?" — has a measurable answer.** Without retrieval priming, AI agents either fall back to raw Angular signals or invent a fake API. Both failure modes are observable here.

2. **The AI-discoverability investment is the right strategic bet.** `llms.txt`, agent skills shipped in npm tarballs, `.cursorrules` / `CLAUDE.md` / `copilot-instructions.md` templates — all of these exist precisely to short-circuit this hallucination. The next benchmark run should measure their effect quantitatively.

3. **Scoring methodology needs hardening.** The current pattern-match scoring auto-favors libraries with thin pattern specs. Compile + behavior tests (currently TODO) are needed for a defensible scorecard.

4. **Free-tier OpenRouter is not sufficient for serious benchmarking.** Three of four free models were rate-limited by upstream providers across the entire run window. The full scorecard requires paid OpenRouter credits or direct provider API keys.

## Next steps

### Immediate (no extra cost)

- **Re-design the scoring** with equalized pattern strictness across all 5 libraries.
- **Add the compile harness** — drop generated code into a stock Angular workspace with each library installed; run `tsc --noEmit`; record diagnostics.
- **Add the behavior harness** — run the prompt's `behavior_test` block in a sealed TestBed.

### With paid OpenRouter credits (~$2–10 per full run)

- Re-run with frontier models: `anthropic/claude-opus-4.7`, `openai/gpt-5.4`, `google/gemini-3.1-pro-preview`, `perplexity/sonar-pro-search`.
- Expected outcome: **frontier models still fail SignalTree without retrieval priming**, but at a lower invention rate (they hallucinate fewer fake APIs, fall back to raw signals more often).

### The killer comparison run

Two runs side-by-side with frontier models:

- **Cold**: just the prompt, no retrieval. (Today's run.)
- **Primed**: prompt + `llms.txt` content injected as system context.

The delta is the headline: _"With SignalTree's AI-discoverability surface injected, agents go from ~0% correct to N% correct on first try. No other Angular state library has that scaffolding."_
